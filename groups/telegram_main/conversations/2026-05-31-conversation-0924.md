# Weather

Two free services, no API keys needed.

## wttr.in (primary)

Quick one-liner:

```bash
curl -s "wttr.in/London?format=3"
# Output: London: ⛅️ +8°C
```

Compact format:

```bash
curl -s "wttr.in/London?format=%l:+%c+%t+%h+%w"
# Output: London: ⛅️ +8°C 71% ↙5km/h
```

Full forecast:

```bash
curl -s "wttr.in/London?T"
```

Format codes: `%c` condition · `%t` temp · `%h` humidity · `%w` wind · `%l` location · `%m` moon

Tips:
- URL-encode spaces: `wttr.in/New+York`
- Airport codes: `wttr.in/JFK`
- Units: `?m` (metric) `?u` (USCS)
- Today only: `?1` · Current only: `?0`
- PNG: `curl -s "wttr.in/Berlin.png" -o /tmp/weather.png`

## Open-Meteo (fallback, JSON)

Free, no key, good for programmatic use:

```bash
curl -s "https://api.open-meteo.com/v1/forecast?latitude=51.5&longitude=-0.12&current_weather=true"
```

Find coordinates for a city, then query. Returns JSON with temp, windspeed, weathercode.

Docs: https://open-meteo.com/en/docs

```bash
summarize "/path/to/file.pdf" --model google/gemini-3-flash-preview
summarize "https://youtu.be/dQw4w9WgXcQ" --youtube auto
```

### Installation

```bash
brew install steipete/tap/summarize
```

### Model + keys

Set the API key for your chosen provider:

- OpenAI: `OPENAI_API_KEY`
- Anthropic: `ANTHROPIC_API_KEY`
- xAI: `XAI_API_KEY`
- Google: `GEMINI_API_KEY` (aliases: `GOOGLE_GENERATIVE_AI_API_KEY`, `GOOGLE_API_KEY`)

Default model is `google/gemini-3-flash-preview` if none is set.

### Useful flags

- `--length short|medium|long|xl|xxl|<chars>`
- `--max-output-tokens <count>`
- `--extract-only` (URLs only)
- `--json` (machine readable)
- `--firecrawl auto|off|always` (fallback extraction)
- `--youtube auto` (Apify fallback if `APIFY_API_TOKEN` set)

If the `summarize` binary is not installed, use Web Fetch to retrieve the content and summarise it directly:

1. Fetch the URL content using the WebFetch tool
2. Summarise the key points from the fetched content
3. For YouTube videos, fetch the video page and extract available transcript/description

This fallback works anywhere including inside containers.

*2. Make It Shorter*

*3. Make It Again*
This is where most of the learning happens. As new info arrives and your rough map gets cluttered or wrong, *reorganise it*. The act of regrouping and reconnecting ideas is what cements memory. Most people skip this step entirely.

*Core idea:* Learning requires iterative, imperfect engagement — not passive recording. The mess *is* the process. 🧠

```bash
ynab budgets list
ynab budgets view [id]
ynab budgets settings [id]
```

```bash
ynab accounts list
ynab accounts view <id>
ynab accounts transactions <id>
```

```bash
ynab transactions list

ynab transactions list --fields id,date,amount,memo
```

```bash
ynab payees transactions <id>
```

### Months

**Andy**: It's a structured 3-stage workflow for writing docs *with* Claude, rather than just asking it to write something for you. Here's how it works:

*Stage 1: Context Gathering*
You dump everything you know — background, constraints, stakeholder concerns, related docs, even messy notes. Claude asks clarifying questions until it properly understands the territory. The goal is closing the gap between what you know and what Claude knows.

*Stage 2: Refinement & Structure*
You build the doc section by section:
1. Claude asks what to include in each section
2. Brainstorms 5–20 options for that section
3. You pick what to keep/drop/combine
4. Claude drafts it
5. You refine through surgical edits (without rewriting the whole thing each time)

*Stage 3: Reader Testing*
Before you share it, Claude tests the doc with a "fresh" Claude instance that has no context from your conversation — simulating how a real reader would experience it. It surfaces blind spots and ambiguities you've become blind to as the author.

---

*Why bother with the workflow vs just asking Claude to write it?*
The structured approach catches the things that trip up most AI-written docs — assuming shared context, skipping the "why", being too generic. The reader testing stage in particular is clever: it's easy to write a doc that makes perfect sense to you but confuses everyone else.

> **PREREQUISITE:** Read `../gws-shared/SKILL.md` for auth, global flags, and security rules. If missing, run `gws generate-skills` to create it.

Send an email with one or more file attachments using Gmail's upload endpoint.

## How It Works

1. Build an RFC 822 MIME multipart message with Python's `email` module
2. Upload it via `gws gmail users messages send --upload <file> --upload-content-type "message/rfc822"`

The `+send` helper does not support attachments — this approach uses the raw upload endpoint instead.

## Usage

### Step 1: Build the MIME message

```bash
python3 -c "
import mimetypes, os
from email.mime.multipart import MIMEMultipart
from email.mime.text import MIMEText
from email.mime.base import MIMEBase
from email import encoders

msg = MIMEMultipart()
msg['To'] = 'RECIPIENT'
msg['Subject'] = 'SUBJECT'
msg.attach(MIMEText('BODY_TEXT', 'plain'))

# Repeat this block for each attachment
filepath = '/path/to/file.pdf'
ctype, _ = mimetypes.guess_type(filepath)
if ctype is None:
    ctype = 'application/octet-stream'
maintype, subtype = ctype.split('/')
with open(filepath, 'rb') as f:
    att = MIMEBase(maintype, subtype)
    att.set_payload(f.read())
encoders.encode_base64(att)
att.add_header('Content-Disposition', 'attachment', filename=os.path.basename(filepath))
msg.attach(att)

with open('/tmp/message.eml', 'wb') as f:
    f.write(msg.as_bytes())
"
```

### Step 2: Send via upload endpoint

```bash
gws gmail users messages send \
  --upload /tmp/message.eml \
  --upload-content-type "message/rfc822"
```

```bash
rm /tmp/message.eml
```

# gmail send attachment

> **PREREQUISITE:** Read `../gws-shared/SKILL.md` for auth, global flags, and security rules. If missing, run `gws generate-skills` to create it.

Send an email with one or more file attachments using Gmail's upload endpoint.

## How It Works

1. Build an RFC 822 MIME multipart message with Python's `email` module
2. Upload it via `gws gmail users messages send --params '{"userId": "me"}' --upload <file> --upload-content-type "message/rfc822"`

The `+send` helper does not support attachments — this approach uses the raw upload endpoint instead.

## Usage

### Step 1: Build the MIME message

```bash
python3 -c "
import mimetypes, os
from email.mime.multipart import MIMEMultipart
from email.mime.text import MIMEText
from email.mime.base import MIMEBase
from email import encoders

msg = MIMEMultipart()
msg['To'] = 'RECIPIENT'
msg['Subject'] = 'SUBJECT'
msg.attach(MIMEText('BODY_TEXT', 'plain'))

# Repeat this block for each attachment
filepath = '/path/to/file.pdf'
ctype, _ = mimetypes.guess_type(filepath)
if ctype is None:
    ctype = 'application/octet-stream'
maintype, subtype = ctype.split('/')
with open(filepath, 'rb') as f:
    att = MIMEBase(maintype, subtype)
    att.set_payload(f.read())
encoders.encode_base64(att)
att.add_header('Content-Disposition', 'attachment', filename=os.path.basename(filepath))
msg.attach(att)

with open('/tmp/message.eml', 'wb') as f:
    f.write(msg.as_bytes())
"
```

### Step 2: Send via upload endpoint

```bash
gws gmail users messages send \
  --params '{"userId": "me"}' \
  --upload /tmp/message.eml \
  --upload-content-type "message/rfc822"
```

### Step 3: Clean up

```bash
rm /tmp/message.eml
```

# Weather

Two free services, no API keys needed.

## wttr.in (primary)

Quick one-liner:

```bash
curl -s "wttr.in/London?format=3"
# Output: London: ⛅️ +8°C
```

Compact format:

```bash
curl -s "wttr.in/London?format=%l:+%c+%t+%h+%w"
# Output: London: ⛅️ +8°C 71% ↙5km/h
```

Full forecast:

```bash
curl -s "wttr.in/London?T"
```

Format codes: `%c` condition · `%t` temp · `%h` humidity · `%w` wind · `%l` location · `%m` moon

Tips:
- URL-encode spaces: `wttr.in/New+York`
- Airport codes: `wttr.in/JFK`
- Units: `?m` (metric) `?u` (USCS)
- Today only: `?1` · Current only: `?0`
- PNG: `curl -s "wttr.in/Berlin.png" -o /tmp/weather.png`

## Open-Meteo (fallback, JSON)

Free, no key, good for programmatic use:

```bash
curl -s "https://api.open-meteo.com/v1/forecast?latitude=51.5&longitude=-0.12&current_weather=true"
```

Find coordinates for a city, then query. Returns JSON with temp, windspeed, weathercode.

Docs: https://open-meteo.com/en/docs

# YNAB (You Need a Budget) — Read-Only

Query budget data from YNAB. This installation is **read-only** — you cannot create, update, or delete anything.

## Setup

Authentication is pre-configured via `YNAB_API_KEY` environment variable.

On first use, set the default budget so you don't need to pass a budget ID every time:

```bash
ynab budgets list                  # find the budget ID
ynab budgets set-default <id>      # set it as default
```

## Commands

All commands return JSON. Add `--compact` for minified output. Amounts are in dollars (not milliunits).

### Budgets

```bash
ynab budgets list
ynab budgets view [id]
ynab budgets settings [id]
```

### Accounts

```bash
ynab accounts list
ynab accounts view <id>
ynab accounts transactions <id>
```

### Categories

```bash
ynab categories list
ynab categories view <id>
ynab categories transactions <id>
```

### Transactions

```bash
# List with filters
ynab transactions list
ynab transactions list --account <id> --since <YYYY-MM-DD>
ynab transactions list --approved=false --min-amount 100
ynab transactions list --fields id,date,amount,memo

# Search
ynab transactions search --memo "coffee"
ynab transactions search --payee-name "Amazon"

# View a single transaction
ynab transactions view <id>
```

### Payees

```bash
ynab payees list
ynab payees view <id>
ynab payees locations <id>
ynab payees transactions <id>
```

### Months

```bash

ynab months view <YYYY-MM>
```

### Scheduled Transactions

```bash
ynab scheduled list
ynab scheduled view <id>
```
