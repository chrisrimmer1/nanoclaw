**Auditor:** Principal-level review (read-only, no code modified)
**Scope:** Chris Rimmer's *customisation layer* + *operational/security posture* of running
"Claudette" (assistant name `Andy` in config) unattended on the Mac mini. Upstream

## Executive Summary

**Health grade (Chris's layer + ops posture): B.**
The fork is unusually well-engineered for a personal automation: the upstream container-isolation
and credential-proxy design is sound, Chris's additions (Apple Container networking, gws ADC
mounting, file-backed keyring, the silent-failure alerting + external heartbeat watchdog) are
careful, defensively coded, and tested. It loses the A because two real secrets (`GITHUB_TOKEN`,
`YNAB_API_KEY`) are injected into the agent container as **plaintext environment variables**,
directly bypassing the credential-isolation model the rest of the system goes to great lengths to
enforce; and because the main agent has **read-write access to every repo under
`~/python_projects`** with `bypassPermissions`, giving a single prompt-injection a large blast
radius. Neither is catastrophic for a 1:1 private chat that only Chris can reach, but they are the
two things worth fixing.

**Top 3 risks**
1. `GITHUB_TOKEN` + `YNAB_API_KEY` handed to the container in cleartext env — the agent can read,
   exfiltrate, or (for GitHub) *write* with them; the read-only ynab shim is bypassable; the
   isolation SECURITY.md advertises for Anthropic creds does **not** apply to these.
   (`src/container-runner.ts:293-307`)
2. Large blast radius: main group mounts all of `~/python_projects` **read-write** +
   `permissionMode: 'bypassPermissions'`, so an injected instruction can rewrite any of Chris's
   project repos. (`~/.config/nanoclaw/mount-allowlist.json`; `agent-runner/src/index.ts:413`)
3. Outdated transitive deps with known CVEs (`ws` high, `yaml` moderate) in a long-lived
   unattended process. (`npm audit`)

**Top 3 opportunities**
1. Route GitHub/YNAB through the existing credential-proxy pattern (or a per-call token-minting
   wrapper) instead of raw env — closes risk #1 with the architecture already present.
2. Scope the main mount to just the repos the agent actually needs, and/or set
   `nonMainReadOnly`-style discipline for `~/python_projects` — shrinks risk #2.
3. Add a real CI test job that runs on Chris's fork (current `ci.yml` only runs on PRs, which a
   solo committer-to-`main` never opens) + `npm audit` gate — restores the safety net the test
   suite was built to provide.

## Repo Map

**Purpose.** Upstream NanoClaw is an open-source "AI assistant that runs agents securely in their
own containers" (TypeScript, Node 22). Chris has forked it (`github.com/chrisrimmer1/nanoclaw`,
upstream `qwibitai/nanoclaw`) and customised it to run a personal Telegram bot ("Claudette",
internal name `Andy`) unattended on his Mac mini via launchd, talking to one private Telegram
chat.

**Stack.** Node.js + TypeScript (ESM), `better-sqlite3`, `grammy` (Telegram), `pino` logging,
`zod`, `vitest`. Runtime: **Apple Container** (Chris switched from Docker — see below). Agents run
the `@anthropic-ai/claude-agent-sdk` inside the container.

**Architecture sketch.** Single host Node process (`src/index.ts`) ← channels self-register
(Telegram) → message routed → host spawns an ephemeral Apple Container per group
(`src/container-runner.ts`) → inside, `container/agent-runner/src/index.ts` drives the Claude Agent
SDK → tool calls back to host via file-based IPC (`/workspace/ipc`). Real Anthropic credentials
never enter the container: the host runs a **credential proxy** (`src/credential-proxy.ts`) that
injects auth headers. Each group has an isolated `data/sessions/<group>/.claude/`.

**Key directories**
| Path | What | Layer |
|------|------|-------|
| `src/` | Host orchestrator, channels, IPC, container runner, proxy, mount security | mostly `[UPSTREAM]`, heavily diverged |
| `src/container-runtime.ts` | Apple Container networking abstraction | `[CHRIS'S LAYER]` (his switch from Docker) |
| `container/Dockerfile`, `container/agent-runner/` | Agent image + in-container runner | `[UPSTREAM]` + Chris's tooling adds |
| `container/skills/` | Agent-facing skills (weather, ynab, granola, agent-github, gws, whats-live…) | `[CHRIS'S LAYER]` |
| `container/ynab-readonly.sh` | Read-only ynab CLI shim | `[CHRIS'S LAYER]` |
| `groups/telegram_main/` | Per-group memory, formatting rules, archived conversations | `[CHRIS'S LAYER]` |
| `launchd/`, `scripts/health-watchdog.sh` | Unattended-run service + external watchdog | `[CHRIS'S LAYER]` |
| `docs/` | SECURITY.md, SPEC.md (upstream), APPLE-CONTAINER-NETWORKING.md (Chris) | mixed |
| `repo-tokens/` | Upstream's GitHub Action for repo "token" badges — **unused by Chris** | `[UPSTREAM]` (dead weight here) |

**Upstream vs Chris's layer (from `git log 451a8dc..HEAD`).** Chris's commits cluster around:
Telegram voice transcription (AssemblyAI), Google Workspace CLI + ADC auth into containers,
GitHub CLI in container, the Apple-Container networking migration, container skills, the
alerting + external watchdog, and per-group memory/skill syncing.

**Surprises.**
- The repo ships a **`.heartbeat` + external watchdog** that notifies Chris's phone if the host
  dies/wedges — genuinely good ops hygiene, better than most hobby setups.
- The agent runs with `permissionMode: 'bypassPermissions'` and `allowDangerouslySkipPermissions:
  true` (`agent-runner/src/index.ts:413-414`) — deliberate (the container *is* the sandbox), but
  it means the mount scope is the *only* thing standing between an injection and the filesystem.
- `data/nanoclaw.db` is 0 bytes; registered groups appear to live in the SQLite/state but the DB
  file is empty on disk — worth confirming state isn't silently being kept only in memory.
- The in-container agent-runner **source is mounted read-write per group and recompiled on every
  start** (`container-runner.ts:186-205`) so the agent can rewrite its *own* runner — powerful and
  intentional, but it's a self-modification surface (scoped to that group's container, re-copied
  from host only if absent).

## Audit Report (findings by dimension, severity-sorted)

### Security

**S1 — Real secrets injected into the container as plaintext env. [CHRIS'S LAYER] — High**
*What:* `GITHUB_TOKEN`, `GH_REPO`, and `YNAB_API_KEY` are read from `.env` and passed via `-e`
into the agent container for **every** run.
*Where:* `src/container-runner.ts:293-307`.
*Why it matters:* The whole point of the credential proxy (SECURITY.md §5) is that "agents cannot
discover real credentials — not in environment, stdin, files, or `/proc`." These two secrets defeat
that: any agent turn can `echo $GITHUB_TOKEN` / `printenv` and exfiltrate them. The GitHub token
(`gho_…`/`ghp_…`, confirmed present, not redacted-out) is **write-capable** — the `agent-github`
skill's "NOT allowed" list (`container/skills/agent-github/SKILL.md:68-79`) is advisory prose the
model can ignore, not an enforced boundary. So a successful prompt-injection in the chat can push
commits, merge PRs, comment, etc. on whatever `GH_REPO` (and, depending on token scope, other
repos) the token can reach. *Fact:* env injection bypasses the proxy. *Judgment:* this is the
single highest-leverage fix.

**S2 — ynab read-only shim is bypassable because the raw token is in the env. [CHRIS'S LAYER] — Medium**
*What:* `container/ynab-readonly.sh` blocks write subcommands of the `ynab` CLI, but the real
`YNAB_API_KEY` is in the container environment (S1).
*Where:* `container/ynab-readonly.sh`; `src/container-runner.ts:294-298`.
*Why it matters:* The shim only guards the *wrapped binary*. With the token in `$YNAB_API_KEY`,
the agent can `curl -X POST https://api.ynab.com/...` directly and perform writes the shim was
meant to prevent. The control is real against accidental CLI misuse but not against an adversarial
prompt. *Judgment:* lower severity than S1 because YNAB blast radius is "mess up a budget," not
"push code / leak a token," but it shows the shim gives a false sense of containment.

**S3 — Main agent has read-write to all of `~/python_projects` under bypassPermissions. [CHRIS'S LAYER] — Medium/High**
*What:* The external mount-allowlist grants `allowReadWrite: true` on
`/Users/chrisrimmer/python_projects` (the whole tree), and the agent runs with permissions
bypassed.
*Where:* `~/.config/nanoclaw/mount-allowlist.json`; agent perms `agent-runner/src/index.ts:413`.
*Why it matters:* Blast radius of any injection = every project repo on the machine (and via S1's
GitHub token, their remotes). The read-only-project-root protection in SECURITY.md §2 protects
*nanoclaw's own* code, not the additional mounts. *Mitigant (fact):* `nonMainReadOnly: true` does
force non-main groups to read-only, and there is currently only one (main) group, so the
read-write surface is "only" reachable by Chris's own chat. *Judgment:* fine today; the moment a
second group is ever added, re-confirm scope. Even for main, narrowing the allowlist to the repos
the agent actually touches would shrink risk cheaply.

**S4 — Sender allowlist defaults to `allow: '*'` and no `sender-allowlist.json` exists. [UPSTREAM default, CHRIS'S config gap] — Low (today)**
*What:* With no `~/.config/nanoclaw/sender-allowlist.json`, every sender in a registered chat is
allowed to trigger the agent.
*Where:* `src/sender-allowlist.ts:17-21` (`DEFAULT_CONFIG`); confirmed file absent.
*Why it matters:* For the current single **private 1:1** chat (`tg:7076872214`, registered
`--is-main --no-trigger-required`) this is safe — only Chris can message it, and Telegram's bot
token is the real gate (unregistered chats are dropped, `telegram.ts:184-191`). It becomes a
genuine risk only if Chris ever registers a *group* chat, where any member would then trigger the
unattended agent. *Judgment:* note-only now; make it a checklist item before adding any group.

**S5 — Telegram bot token printed into URLs / lives in `.env` read at runtime. [UPSTREAM] — Low**
*What:* Voice handling builds `https://api.telegram.org/file/bot${this.botToken}/...`
(`telegram.ts:253`).
*Why it matters:* Token appears in any code path that logs that URL; it's the standing credential
for the bot. It is not currently logged at info level, and `.env` is correctly git-ignored
(verified: `.env` is **not** tracked; global ignore + repo `.gitignore` both cover it). *Judgment:*
acceptable; just don't add debug logging of that URL.

**S6 — Credential proxy binds `0.0.0.0` on the Apple Container path. [UPSTREAM, Chris's runtime] — Low (handled)**
*What:* On Apple Container, the proxy binds `0.0.0.0` because the bridge gateway IP doesn't exist
until the first container starts.
*Where:* `src/container-runtime.ts:86-117`; gating in `credential-proxy.ts:48-60,166-176`.
*Why it matters:* `0.0.0.0` is reachable from the LAN, not just containers. *Mitigant (fact):*
access is gated by `getProxyAllowedCidr()` to loopback + the container subnet, and requests from
other sources get `403`. This is a thoughtful, correct mitigation. *Judgment:* documented and
handled; flagged only so it's on the record that the bind is wide-by-necessity, narrow-by-filter.

### Architecture & design

**A1 — `src/container-runner.ts` is a god-file (~780 lines) doing mounts, secret-passing, arg-building, process lifecycle, timeout logic, and log formatting. [UPSTREAM, Chris-extended] — Low**
*Why it matters:* The secret-injection block (S1) is buried mid-file in `buildContainerArgs`,
which is exactly why it's easy to miss that it bypasses the proxy. Splitting mount/secret policy
into its own module would make the security-relevant surface reviewable in isolation. *Judgment:*
maintainability, not a bug; low priority since it's largely upstream code Chris shouldn't own.

**A2 — Self-modifiable agent-runner source mounted read-write. [UPSTREAM] — Low**
Intentional design (per-group customisation, `container-runner.ts:183-205`). Noted as a surface,
not a defect; scoped per-group and re-seeded from host only when absent.

### Dependencies

**D1 — Known-CVE transitive deps in a long-running unattended process. [UPSTREAM deps, Chris runs it] — Medium**
`npm audit`: `ws` (High — uninitialized memory disclosure / DoS) and `yaml` (Moderate — stack
overflow on deeply nested input). Both have fixes available via `npm audit fix`. `yaml` is a direct
dep (`package.json`); `ws` is transitive (via grammy/SDK). For an always-on bot exposed to attacker
-influenceable input (chat text, web fetches), patching is worth doing. *Judgment:* low effort,
real value.

### Testing

**T1 — CI never runs on Chris's actual workflow. [CHRIS'S LAYER / config] — Medium**
*What:* `ci.yml` triggers only on `pull_request` to `main`. Chris commits **straight to `main`**
(his convention) and is a solo operator, so CI effectively never runs on his pushes.
*Where:* `.github/workflows/ci.yml` (`on: pull_request`).
*Why it matters:* There's a real test suite (`vitest`, many `*.test.ts`, format + typecheck), but
nothing enforces it on the way changes actually land. Regressions in the diverged host code
(routing, sender-allowlist, container-runner) ship unchecked. *Judgment:* add a `push: [main]`
trigger or a local pre-push hook running `npm test`.

**T2 — Husky pre-commit exists but only formats. [UPSTREAM] — Low**
`.husky/pre-commit` — worth confirming it runs typecheck/tests, not just prettier, given T1.

### DevEx & operations

**O1 — Strong: external heartbeat watchdog + silent-failure alerting. [CHRIS'S LAYER] — Strength**
`scripts/health-watchdog.sh` + the alerting commit give real "is it alive / did it silently fail"
coverage with phone notifications and cooldowns. This is the best part of the ops layer.

**O2 — `data/nanoclaw.db` is 0 bytes. [investigate] — Low**
Worth a glance to confirm registered-group/state persistence is actually durable and not held only
in process memory (a crash-restart that loses registration would be a quiet outage). *Fact:* file
is 0 bytes as of audit; *judgment:* needs a 5-minute confirm, not necessarily a fix.

**O3 — launchd plist hardcodes `ASSISTANT_NAME=Andy`. [CHRIS'S LAYER] — Low/cosmetic**
The product is called "Claudette" in Chris's notes but the runtime/persona name is `Andy`
(`launchd/com.nanoclaw.plist`; default in `config.ts:12`). Harmless, but a naming drift that could
confuse future-Chris.

### Documentation

**Doc1 — SECURITY.md is accurate for Anthropic creds but silent on the GitHub/YNAB env exception. [CHRIS'S LAYER] — Medium**
SECURITY.md §5 categorically states agents can't see credentials in the environment. That's true
for Anthropic but **false** for `GITHUB_TOKEN`/`YNAB_API_KEY` (S1). The doc should either carve out
the exception explicitly or (better) the code should be fixed so the doc stays true. Stale-doc-
contradicting-code is exactly the failure mode to avoid. *Judgment:* fix code, then the doc is
correct as-is.

**Doc2 — `setup-log.md` says runtime is Docker; it's now Apple Container. [CHRIS'S LAYER] — Low**
`setup-log.md` §4/§9 record Docker; the live system runs Apple Container (`container-runtime.ts`,
`APPLE-CONTAINER-NETWORKING.md`). Minor historical drift; APPLE-CONTAINER-NETWORKING.md is the
current truth.

### Strengths (what the repo does well)
- Credential proxy for Anthropic auth is a genuinely strong isolation primitive, correctly
  implemented (header stripping, hop-by-hop removal, CIDR gating, OAuth vs API-key modes).
- Mount security: external tamper-proof allowlist, symlink resolution before validation, container-
  path traversal checks, `nonMainReadOnly`. Well-tested.
- Per-group session/IPC isolation prevents cross-group disclosure.
- Chris's Apple Container networking work is careful (lazy network discovery, hard-fail with clear
  operator guidance, no silent guessing).
- Real, useful tests across the diverged host modules; clean prettier/typecheck discipline.
- Operational maturity: heartbeat, external watchdog, phone alerts on silent failure.

## Improvement Strategy (themes)

**Theme 1 — Close the credential-isolation leak (explains S1, S2, Doc1).**
*Target state:* No long-lived write-capable secret sits in the container environment. GitHub and
YNAB access, if needed at all, go through the same "host holds the secret, container gets a
scoped/placeholder path" pattern the Anthropic proxy already proves out — or through host-side
tool wrappers invoked over IPC. *Principle:* the container is untrusted; secrets live host-side and
are injected per-request, never handed over wholesale. *Done =* `printenv` in a container reveals
no real GitHub/YNAB token; SECURITY.md's environment claim is true without caveats.
*Trade-off / what NOT to do:* don't try to "lock down" the agent with prompt instructions
(`agent-github` "NOT allowed" list) — that's not a security boundary. Don't over-engineer a full
proxy for YNAB if the simpler answer is "the read-only use case doesn't need a write-capable token
at all" (consider a read-scoped token).

**Theme 2 — Shrink blast radius (explains S3, S4).**
*Target state:* The agent can write only to the specific repos/dirs it genuinely operates on, and
the sender-allowlist is explicitly configured before any non-1:1 chat is ever registered.
*Principle:* least privilege on mounts; default-deny on senders for anything but the private chat.
*Done =* mount-allowlist lists only needed roots; a `sender-allowlist.json` exists (even if it just
pins the main chat) so the default isn't silently `*`. *Trade-off:* narrowing the mount costs some
convenience (agent can't touch a new project until you add it) — acceptable for the safety gain;
keep `~/python_projects` broad only if Chris consciously wants that.

**Theme 3 — Make the safety net actually fire (explains T1, T2, D1).**
*Target state:* Tests + typecheck + `npm audit` run on the path changes actually take (push to
`main` or pre-push), and CVE deps are patched. *Principle:* a test suite that never runs on your
workflow is documentation, not protection. *Done =* a green check appears on Chris's own pushes;
`npm audit` is clean or consciously waived.

**Theme 4 — Keep docs honest (Doc1, Doc2, O3).**
*Target state:* SECURITY.md has no false categorical claim; setup-log reflects Apple Container;
naming (Claudette/Andy) is reconciled or annotated. *Principle:* docs that contradict code are
worse than no docs. Mostly falls out of Theme 1.

## Open Questions
1. **GitHub token scope.** Is the `.env` `GITHUB_TOKEN` a fine-grained, single-repo, read-only
   token, or a classic broad-scope PAT? This determines whether S1 is "leak a low-value read
   token" or "leak the keys to all your repos." (Could not inspect scope read-only without using
   the token.)
2. **Does the agent actually need GitHub *write* in-container?** If the real use is "read issues/PRs,
   maybe comment," a read-or-comment-scoped token removes most of S1's bite.
3. **DB persistence:** is `data/nanoclaw.db` genuinely empty because state lives elsewhere, or is
   registration kept only in memory between restarts? (O2/TK8.)
4. **Future group chats:** any plan to register a non-private (group/supergroup) Telegram chat? If
   yes, S4 (sender allowlist) and S3 (mount scope) move from "note" to "must-do-first."
5. **`repo-tokens/` directory:** confirmed unused by Chris's deployment — safe to leave as upstream
   dead weight, or worth deleting from the fork to reduce confusion?
